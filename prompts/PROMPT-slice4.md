## ПРОМПТ 4a — Контейнеризация + Kubernetes + Helm

Ты — platform/DevOps-инженер. В репозитории `Lingua` готовы Срезы 1–3, всё пока
запускается через docker-compose. **Прочитай существующий код и compose-файлы.**
Цель — перевести систему на Kubernetes, не ломая dev-флоу через compose.

Построй:

- Production-grade `Dockerfile` для каждого сервиса (multi-stage, не-root, минимальные образы; для Python — отдельные).
- Kubernetes-манифесты + **Helm-чарты** (чарт на сервис + umbrella-чарт окружения).
- Локальный кластер: **kind** или **k3d**; инструкция поднять всё локально в k8s.
- Конфиги/секреты (ConfigMap/Secret), ingress (Traefik или NGINX).
- Зависимости (PostgreSQL, Redis, Kafka (KRaft), Keycloak, MinIO) — через Helm-чарты/операторы.
- Kubernetes 1.36.x, Helm 3.x.

DoD: `helm install` (или скрипт) поднимает весь стек в локальном k8s; сквозной флоу из Срезов 1–3 работает в кластере; docker-compose остаётся рабочим для dev.
НЕ-цели: OpenTelemetry/Prometheus (4b), gRPC (4c), CI/CD/Terraform (4d).
Сначала прочитай репозиторий и покажи план.

## ПРОМПТ 4b — Observability (OpenTelemetry + Prometheus + Grafana + Loki)

Ты — platform-инженер. В репозитории `Lingua` готовы Срезы 1–3 и k8s/Helm (4a).
**Прочитай существующий код.** Цель — сквозная наблюдаемость event-driven системы.

Построй:

- **OpenTelemetry**-инструментацию всех сервисов (TS через `libs/observability`, Python отдельно): трейсы по HTTP и по Kafka.
- **Проброс trace-контекста через Kafka** (W3C traceparent в заголовках сообщений), чтобы трейс не рвался на асинхронных границах — особенно по флоу `review.completed` и петле разговора.
- OTel Collector; экспорт трейсов; **Prometheus** (метрики) + **Grafana** (дашборды) + **Loki** (логи).
- Готовые дашборды: латентность сервисов, лаг консьюмеров Kafka, сквозной трейс одного повторения и одного разговора.

DoD: один пользовательский сценарий (повторение карточки и разговор) виден как **единый сквозной трейс** через несколько сервисов и через Kafka; метрики и логи собираются в Grafana.
НЕ-цели: gRPC (4c), CI/CD/Terraform (4d).
Сначала прочитай репозиторий и покажи план.

## ПРОМПТ 4c — gRPC для внутренних вызовов + Schema Registry

Ты — backend-инженер. В репозитории `Lingua` готовы Срезы 1–3, k8s (4a), observability (4b).
**Прочитай существующий код.** Цель — типизировать синхронные межсервисные вызовы.

Построй:

- **gRPC (Protobuf)** для внутренних sync-вызовов сервис↔сервис там, где сейчас внутренний REST. Proto-контракты — в `libs/contracts`.
- К фронту оставить REST/GraphQL + WebSocket (gRPC из браузера напрямую нельзя — без gRPC-Web).
- Включить **Schema Registry** для Kafka (Avro/Protobuf), вынести проверку совместимости схем в CI.
- Сохранить кросс-языковую генерацию (TS + Python) из proto/схем.

DoD: ключевые внутренние вызовы идут по gRPC с типизированными контрактами; события Kafka проходят через Schema Registry; edge остаётся REST/GraphQL/WS; всё работает в k8s.
НЕ-цели: CI/CD-пайплайн целиком и Terraform (4d) — кроме шага проверки схем, который вводится здесь.
Сначала прочитай репозиторий и покажи план.

## ПРОМПТ 4d — CI/CD (GitHub Actions) + IaC (Terraform)

Ты — DevOps-инженер. В репозитории `Lingua` готовы Срезы 1–3 и инфра-слои 4a–4c.
**Прочитай существующий код и Helm-чарты.** Цель — автоматизировать сборку/деплой и инфру.

Построй:

- **GitHub Actions** на базе **`nx affected`**: lint → test → build → docker build/push только затронутых проектов; кэш Nx.
- Шаг деплоя через Helm в кластер (staging); проверка совместимости Kafka-схем (из 4c) в пайплайне.
- **Terraform** (или OpenTofu): managed k8s-кластер + managed PostgreSQL/Kafka под живое демо; параметризация окружений.
- Документация: как поднять демо-окружение с нуля.

DoD: пуш в основную ветку прогоняет affected-пайплайн и деплоит в staging; `terraform apply` поднимает демо-инфру; README с инструкцией демо.
НЕ-цели: нет — это финальный слой.
Сначала прочитай репозиторий и покажи план.
